Для oplot (телеметрия, save-история, аналитика балансa) или любого нормального проекта — пользуйся свободно.  *Проверенные паттерны для time-series в SQLite*  *1. Hot / Warm / Cold через ATTACH* Главная идея: одна «горячая» БД для свежих данных + архивные файлы по периодам, цепляешь через `ATTACH DATABASE`. ``` main.db          -- последние N дней, индексы, FK archive_2026Q1.db archive_2026Q2.db ``` Запросы к свежему — быстрые. Исторические — `ATTACH ... AS arc; SELECT ... FROM arc.events WHERE ...`. Архивные файлы можно держать в read-only режиме (`PRAGMA query_only=1`), бэкапить отдельно, дропать целиком при retention.  *2. Rolling window + батчевое перемещение* Крон/триггер раз в сутки: ```sql BEGIN IMMEDIATE; INSERT INTO arc.events SELECT * FROM main.events WHERE ts < :cutoff; DELETE FROM main.events WHERE ts < :cutoff; COMMIT; ``` Важно: батчами по 10–50k строк, не одним DELETE на миллион — иначе WAL пухнет и блокировки.  *3. Партиционирование «руками» через VIEW* SQLite не умеет нативные партиции, но эмулируется: ```sql CREATE VIEW events_all AS   SELECT * FROM main.events   UNION ALL SELECT * FROM arc1.events   UNION ALL SELECT * FROM arc2.events; ``` Пишешь только в `main.events`, читаешь из `events_all` когда нужна история. Планировщик прунит по WHERE ts.  *4. Компакция* - `PRAGMA journal_mode=WAL` + `PRAGMA wal_autocheckpoint=1000` — обязательно. - После крупных DELETE — `PRAGMA incremental_vacuum` (если включён `auto_vacuum=INCREMENTAL` *с самого создания БД*, иначе бесполезно). - Раз в неделю/месяц — `VACUUM INTO 'main_compact.db'` в отдельный файл, потом атомарный swap. Не блокирует читателей в отличие от обычного `VACUUM`. - Для агрегатов: rollup-таблицы (`events_hourly`, `events_daily`) пишутся триггером или джобой, raw держится только N дней.  *5. Referential integrity через границы* FK *между* attached БД не работают — SQLite их не проверяет cross-database. Варианты: - a) Архивируй замкнутыми поддеревьями: переносишь parent + всех его children одной транзакцией, FK в архивной БД сохраняются локально. - b) Денормализуй при архивации: материализуй нужные поля родителя прямо в архивную строку (write-once данные, дешево по месту). - c) Soft-archive: не двигай строки, добавь `archived_at` + частичный индекс `WHERE archived_at IS NULL` для горячих запросов. FK целы, но файл растёт — годится до десятков млн строк.  *6. Что обычно ломается* - Индексы на `ts DESC` — без них rolling DELETE медленный. - Отсутствие `PRAGMA foreign_keys=ON` (по умолчанию OFF в SQLite, многие забывают). - `auto_vacuum` нельзя включить задним числом без полного дампа/рестора — решай на старте. - `ATTACH` лимит — 10 БД по умолчанию (`SQLITE_MAX_ATTACHED`), для много-квартальной истории либо ребилди под больший лимит, либо объединяй старые кварталы в годовые архивы.  *Дефолтный рецепт под «не знаю масштаб»* WAL + auto_vacuum=INCREMENTAL с самого начала + одна main.db с retention 30–90 дней + ночной джоб, который переносит старое в `archive_YYYYMM.db` батчами по 20k + rollup-таблицы для дашбордов + `VACUUM INTO` swap раз в месяц. Покрывает 95% кейсов до сотен млн строк.  Если скажешь реальный объём (строк/день, размер строки, паттерн чтения — последние сутки vs полная история) — подскажу конкретные числа для cutoff и батча.
