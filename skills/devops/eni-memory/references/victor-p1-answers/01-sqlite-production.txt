Кароч, по делу. SQLite в проде — это в основном 6 PRAGMA + дисциплина с транзакциями. Ничего магического. *PRAGMA-набор (выставлять на каждом коннекте, кроме journal_mode — он persistent):* ``` PRAGMA journal_mode = WAL;          -- writers не блокируют readers PRAGMA synchronous  = NORMAL;       -- с WAL безопасно; FULL — оверкилл, OFF — опасно PRAGMA foreign_keys = ON;           -- ВАЖНО: не persistent, ставить на каждом connect PRAGMA busy_timeout = 5000;         -- ждать 5с вместо мгновенного SQLITE_BUSY PRAGMA temp_store   = MEMORY; PRAGMA cache_size   = -65536;       -- 64 MiB на коннект (минус = KiB) PRAGMA mmap_size    = 268435456;    -- 256 MiB memory-mapped I/O PRAGMA wal_autocheckpoint = 1000;   -- checkpoint каждые ~1000 страниц (default ок) ``` *Почему WAL + synchronous=NORMAL — это safe-default:* в WAL-режиме `NORMAL` гарантирует durability на checkpoint и consistency после краша; теряется максимум последняя НЕзачекпоинченная транзакция, БД при этом не повреждается. Это документировано SQLite-авторами как рекомендованный прод-режим. `FULL` дает durability на каждый commit ценой ~2-5x write latency. *Python — connection helper (stdlib `sqlite3`, без ORM):* ```python import sqlite3, threading from contextlib import contextmanager  DB_PATH = "app.db"  def _connect() -> sqlite3.Connection:     conn = sqlite3.connect(         DB_PATH,         timeout=30.0,         isolation_level=None,        # manual BEGIN/COMMIT — никакой магии         check_same_thread=False,     # мы сами следим за thread-safety     )     conn.row_factory = sqlite3.Row     conn.executescript("""         PRAGMA journal_mode = WAL;         PRAGMA synchronous  = NORMAL;         PRAGMA foreign_keys = ON;         PRAGMA busy_timeout = 5000;         PRAGMA temp_store   = MEMORY;         PRAGMA cache_size   = -65536;         PRAGMA mmap_size    = 268435456;     """)     return conn  _local = threading.local() def get_conn() -> sqlite3.Connection:     # один коннект на поток — sqlite3 module-level thread-safe, но cursor нет     if not hasattr(_local, "conn"):         _local.conn = _connect()     return _local.conn  @contextmanager def tx(write: bool = False):     conn = get_conn()     conn.execute("BEGIN IMMEDIATE" if write else "BEGIN")     try:         yield conn         conn.execute("COMMIT")     except BaseException:         conn.execute("ROLLBACK")         raise ``` *Главные грабли, на которые ловятся все:*  • `BEGIN IMMEDIATE` для пишущих транзакций. Дефолтный `BEGIN DEFERRED` берет write-lock только на первом INSERT/UPDATE — и если между SELECT и UPDATE влез другой writer, ты ловишь `SQLITE_BUSY` посреди транзакции и должен откатываться и ретраить с нуля. `IMMEDIATE` берет лок сразу → конкурируешь до начала работы, а не после.  • `foreign_keys = ON` — НЕ persistent. Забыл на новом коннекте — FK молча игнорируются. Проверка: `PRAGMA foreign_keys;` должно вернуть 1.  • Только ОДИН writer одновременно. WAL разрешает много readers + 1 writer. Если у тебя multi-process writer — сериализуй через `BEGIN IMMEDIATE` + `busy_timeout`, либо вынеси writes в отдельный процесс/очередь.  • Параметризованные запросы всегда (`?` или `:name`), никаких f-string в SQL.  • `executemany` для bulk insert внутри одной транзакции — разница 100-1000x против commit-per-row. *Crash recovery — что делать:*  Короткий ответ: ничего. WAL восстанавливается автоматически при следующем `sqlite3.connect()` — недозачекпоинченные транзакции либо реплеятся, либо отбрасываются по atomic-commit-протоколу. Файлы `app.db-wal` и `app.db-shm` после краша оставлять на месте — они нужны для recovery.  Что полезно делать руками: ```python def checkpoint(conn):  # форсированный checkpoint, например перед бэкапом     conn.execute("PRAGMA wal_checkpoint(TRUNCATE);")  def integrity_check(conn) -> bool:     row = conn.execute("PRAGMA integrity_check;").fetchone()     return row[0] == "ok" ``` Интегрити-чек гонять при старте приложения или раз в сутки кроном — он читает всю БД, на больших базах не дешевый. *Бэкап (правильный способ, hot, без остановки writers):* ```python def backup(src_path: str, dst_path: str):     src = sqlite3.connect(src_path)     dst = sqlite3.connect(dst_path)     with dst:         src.backup(dst)   # SQLite Online Backup API, атомарно, page-by-page     src.close(); dst.close() ``` Или проще, одной командой: `VACUUM INTO 'backup-2026-06-08.db';` — даёт компактную копию без фрагментации. `cp app.db backup.db` на живой БД — ❌, можно поймать torn write. *Когда SQLite не подходит:* multi-host writes, write-throughput > ~1-5k tx/s sustained, нужен row-level locking без сериализации writers. Для всего остального (включая прод-сервисы с миллионами читателей) — топ.
