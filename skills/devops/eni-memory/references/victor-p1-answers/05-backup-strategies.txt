Кароч, по убыванию надёжности — три уровня, выбирай по тому, на каком FS живёшь и как часто нужны снапшоты.  *1. Filesystem snapshot (самое надёжное, реально атомарно)* Если БД и логи на ZFS / btrfs / LVM-thin — снапшот файловой системы единственный способ получить *гарантированно* консистентный снимок DB+JSONL одной точкой во времени: ``` # ZFS zfs snapshot tank/data@$(date +%Y%m%d-%H%M) zfs send tank/data@... | ...   # вынос наружу  # btrfs btrfs subvolume snapshot -r /data /snaps/$(date +%s)  # LVM-thin lvcreate --snapshot --name snap1 vg/data ``` Дальше уже спокойно копируешь снапшот чем угодно (`restic`, `rsync`, `borg`). SQLite в WAL-режиме переживёт это нормально: при восстановлении просто доиграет WAL.  *2. SQLite Online Backup API + ротация JSONL (если FS обычный ext4/xfs)* Идея: создать чёткую границу «всё до этого момента» одной транзакцией. ``` sqlite3 app.db "PRAGMA journal_mode=WAL;"   # один раз  # атомарный снапшот: sqlite3 app.db <<'SQL' BEGIN IMMEDIATE;             -- забираем write-lock .shell mv audit.jsonl audit-$(date +%s).jsonl && touch audit.jsonl COMMIT; SQL sqlite3 app.db ".backup '/backup/app-$(date +%s).db'" ``` Ключ — `BEGIN IMMEDIATE` + ротация JSONL внутри транзакции: всё, что попало в БД к моменту коммита, точно соответствует закрытому файлу лога. Новые записи уже идут в новый файл.  Или проще, без шелла внутри sqlite: - application-side: взять writer-lock приложения → `fsync` JSONL → `rename` его в `audit-TS.jsonl` → `sqlite3_backup_*` (или `VACUUM INTO '/backup/app-TS.db'`) → отпустить лок.  `VACUUM INTO` удобнее `.backup` тем, что сразу даёт дефрагментированную копию одним вызовом и работает в одной транзакции: ``` sqlite3 app.db "VACUUM INTO '/backup/app-$(date +%s).db';" ```  *3. Continuous replication — Litestream* Если нужен не «снапшот раз в N часов», а *near-zero RPO*: - `litestream` стримит WAL-фреймы SQLite в S3/MinIO/локальный диск непрерывно. Восстановление на любую секунду. - JSONL логи отдельно гонишь тем же `restic`/`rclone` каждые N минут. - Консистентность между ними тогда даётся не атомарностью, а тем что в JSONL пишется `(ts, sqlite_rowid/txn_id)` — при восстановлении отрезаешь хвост лога по последнему txn, который доехал в БД.  *Что я бы реально поставил* - ZFS/btrfs под `/var/lib/app` + `zfs-auto-snapshot` (или `btrbk`) каждые 15 мин, ежедневный `restic` снапшота наружу. Это даёт атомарность + offsite + дедуп бесплатно. - Если FS поменять нельзя — `VACUUM INTO` + ротация JSONL внутри `BEGIN IMMEDIATE`, дальше `restic` папки с бэкапами. - Поверх в любом случае проверка: после каждого бэкапа `sqlite3 backup.db 'PRAGMA integrity_check;'` и `wc -l` по JSONL — иначе однажды обнаружишь, что бэкапил пустоту полгода.  *Чего избегать* - `cp app.db` на живой БД без WAL-checkpoint — почти гарантированно битый файл. - `tar` по каталогу с открытой БД без снапшота FS — то же самое. - Бэкап только `.db` без `.db-wal`/`.db-shm` если копируешь файлами — потеряешь незачекпоинченные транзакции.
