Алекс, по сути вопрос generic — отвечаю по-честному. В SQLite для undo последней операции есть 5 рабочих паттернов, и выбор зависит от того, *в какой момент* ты хочешь откатывать и *насколько глубоко*.

*1. SAVEPOINT / ROLLBACK TO SAVEPOINT — самый дешёвый, если undo нужен только внутри одной сессии*  ```sql SAVEPOINT before_op; -- ... INSERT/UPDATE/DELETE ... -- передумали: ROLLBACK TO SAVEPOINT before_op; RELEASE SAVEPOINT before_op; ``` SAVEPOINT'ы вложенные, именованные, бесплатные. Минус: живут только до COMMIT внешней транзакции. Для «юзер закрыл приложение и через час нажал Ctrl+Z» — не подходит.  *2. Канонический undo/redo на триггерах (официальный паттерн sqlite.org)*  Идея: на каждую таблицу вешаешь 3 триггера (AFTER INSERT/UPDATE/DELETE), которые пишут *обратную* SQL-команду в служебную таблицу `undolog(seq INTEGER PRIMARY KEY, sql TEXT)`. Undo = вычитать строки из последнего «барьера» и выполнить их EXEC'ом в обратном порядке; redo симметрично.  ```sql CREATE TEMP TABLE undolog(seq INTEGER PRIMARY KEY, sql TEXT);  CREATE TRIGGER tbl_it AFTER INSERT ON tbl BEGIN   INSERT INTO undolog VALUES(NULL, 'DELETE FROM tbl WHERE rowid='||new.rowid); END; CREATE TRIGGER tbl_ut AFTER UPDATE ON tbl BEGIN   INSERT INTO undolog VALUES(NULL,     'UPDATE tbl SET a='||quote(old.a)||', b='||quote(old.b)||     ' WHERE rowid='||old.rowid); END; CREATE TRIGGER tbl_dt BEFORE DELETE ON tbl BEGIN   INSERT INTO undolog VALUES(NULL,     'INSERT INTO tbl(rowid,a,b) VALUES('||old.rowid||','||     quote(old.a)||','||quote(old.b)||')'); END; ``` Плюсы: работает прозрачно для любого кода, который пишет в БД, не только для «знающего» слоя. Минусы: триггеры надо генерить per-column (есть скрипты-генераторы), `undolog` растёт — нужно подрезать.

*3. Session extension / changesets — самый «взрослый» вариант*  SQLite из коробки умеет записывать *changeset* — бинарный diff всех изменений в сессии. Потом его можно *invert* и *apply* обратно — это и есть undo, без ручных триггеров.  API (C, но есть обёртки): `sqlite3session_create`, `sqlite3session_attach`, `sqlite3session_changeset`, `sqlite3changeset_invert`, `sqlite3changeset_apply`. Python: `apsw` полностью оборачивает (`apsw.Session`, `apsw.Changeset`); в стандартном `sqlite3` их нет.  ```python import apsw conn = apsw.Connection("app.db") ses = apsw.Session(conn, "main") ses.attach()           # все таблицы ses.attach("users")    # или конкретная # ... делаем изменения ... cs = ses.changeset()                # bytes undo = apsw.Changeset.invert(cs)    # bytes — хранишь в stack’е # чтобы откатить: apsw.Changeset.apply(undo, conn) ``` Плюсы: работает без schema changes, хранит только *фактически* изменённые строки, conflict-handler для случая «строку уже поменяли после». Минусы: не ловит DDL, требует PRIMARY KEY на таблице.  *4. Application-level command/event log (event sourcing lite)*  Не хранишь «обратный SQL» — хранишь *семантическую* операцию + её inverse на уровне домена: ```sql CREATE TABLE op_log(   id INTEGER PRIMARY KEY,   ts INTEGER, user TEXT,   op TEXT,          -- 'add_entry' / 'edit_entry' / ...   payload JSON,     -- что меняли   inverse JSON      -- что нужно, чтобы отменить ); ``` Плюсы: undo выражается в терминах бизнес-логики (audit бесплатно), стабилен к миграциям схемы, легко показать юзеру «отменить: добавлено X». Минус: нужно все мутации проводить через единую «командную» точку, иначе пропустишь.  *5. Temporal / versioned rows (soft history)*  Каждый row версионируешь (`valid_from`, `valid_to`, `deleted_at`) и *никогда* не делаешь UPDATE/DELETE по месту — INSERT новой версии. Undo = пометить последнюю версию как tombstone'нутую. Хорошо, если данных мало и история всё равно нужна.

*Как выбрать — коротко:*  | Сценарий | Бери | |---|---| | Undo внутри одного «шага» в одной сессии | SAVEPOINT | | Ctrl+Z в desktop/CRUD-приложении, много таблиц | триггеры + undolog (#2) | | Синхронизация / offline-первый сервис, нужен invert реальных row-changes | sessions/changesets (#3) | | Бизнес-операции с аудитом («юзер добавил запись») | op_log + inverse (#4) | | Нужна полная история / time-travel | versioned rows (#5) |  *Грабли, которые все забывают:* - AUTOINCREMENT / внешние эффекты (FS, HTTP, другая БД) **не откатятся** ни одним из этих способов — это нужно отдельно (compensating actions). - Триггеры (#2) не видят `INSERT OR REPLACE`, `ON CONFLICT REPLACE` и DDL — будь внимателен. - Для undo *после* перезапуска процесса undolog/op_log должны быть персистентными таблицами, не TEMP. - Группируй в«барьеры» (undo step = всё между двумя маркерами), иначе «откатить последнее действие юзера» превращается в «откатить один UPDATE из 17».  Каноничная страница по #2 — https://sqlite.org/undoredo.html, по #3 — https://sqlite.org/sessionintro.html.  Если скажешь, что конкретно за «данная запись юзера» (сингл-табличная? многосвязная? нужна ли redo? история больше чем на 1 шаг?) — подскажу именно под твой кейс.
