*Complete module:* journal_search.py — stdlib + `sqlite3` only, FTS5 smoke-tested locally (insert/update/delete triggers + BM25 + snippets all pass). Key pieces below; design notes at the end.  *Design choices that matter:* • FTS5 uses *external-content* (`content='messages'`) so it stores only the inverted index, not a 2nd copy of your text. • Vector sync is *manual* (embeddings need a model) — only a DELETE trigger auto-cleans the vec table. Backfill is resumable via a `LEFT JOIN ... IS NULL` cursor. • Hybrid fusion defaults to *RRF* (rank-based) because BM25 scores and cosine distances aren't on comparable scales. A weighted min-max variant is included if you prefer score blending.

*1) FTS5 schema + sync triggers* ```sql CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts USING fts5(     content,     content='messages',          -- external content table     content_rowid='id',     tokenize='unicode61 remove_diacritics 2'   -- add leading 'porter' for stemming );  CREATE TRIGGER IF NOT EXISTS messages_ai AFTER INSERT ON messages BEGIN     INSERT INTO messages_fts(rowid, content) VALUES (new.id, new.content); END;  CREATE TRIGGER IF NOT EXISTS messages_ad AFTER DELETE ON messages BEGIN     INSERT INTO messages_fts(messages_fts, rowid, content)     VALUES ('delete', old.id, old.content);          -- external-content delete END;  CREATE TRIGGER IF NOT EXISTS messages_au AFTER UPDATE ON messages BEGIN     INSERT INTO messages_fts(messages_fts, rowid, content)     VALUES ('delete', old.id, old.content);     INSERT INTO messages_fts(rowid, content) VALUES (new.id, new.content); END;  -- one-time backfill of existing rows: INSERT INTO messages_fts(messages_fts) VALUES ('rebuild'); ``` *BM25 + snippet query* (bm25() is more-negative = better, so `ORDER BY score ASC`): ```sql SELECT m.id, m.role, m.created_at,        bm25(messages_fts) AS score,        snippet(messages_fts, 0, '[', ']', ' … ', 12) AS snip FROM messages_fts JOIN messages m ON m.id = messages_fts.rowid WHERE messages_fts MATCH ?      -- 'morning AND run', '"exact phrase"', 'jog*' ORDER BY score ASC LIMIT ?; ```

*2) sqlite-vec schema + KNN* ```sql CREATE VIRTUAL TABLE IF NOT EXISTS messages_vec USING vec0(     message_id INTEGER PRIMARY KEY,     session_id TEXT,                          -- metadata col: filterable inside KNN     embedding  FLOAT[384] distance_metric=cosine   -- use 768 for larger models );  CREATE TRIGGER IF NOT EXISTS messages_vec_ad AFTER DELETE ON messages BEGIN     DELETE FROM messages_vec WHERE message_id = old.id; END; ``` *KNN query* — sqlite-vec needs `k = ?` in the WHERE for the index path; `distance` is cosine distance in [0,2], so similarity = `1 - distance`: ```sql SELECT v.message_id AS id, v.distance, (1.0 - v.distance) AS cosine_sim,        m.content, m.created_at FROM messages_vec v JOIN messages m ON m.id = v.message_id WHERE v.embedding MATCH ?      -- serialize_f32(query_vec)   AND k = ?   -- AND session_id = ?     -- optional metadata pre-filter ORDER BY v.distance; ``` *Vectors* go in as little-endian float32 blobs (~4× smaller than JSON): ```python import struct def serialize_f32(vec):     return struct.pack("%sf" % len(vec), *vec) ```

*Batch backfill (resumable)* — only embeds rows missing a vector, walks by id so it can be re-run safely after a crash: ```python for rows in iter_unembedded(conn, batch_size=256):   # LEFT JOIN messages_vec WHERE v.message_id IS NULL     vectors = embed_fn([r["content"] for r in rows])   # your model -> list[list[float]]     conn.executemany(         "INSERT INTO messages_vec(message_id, session_id, embedding) VALUES (?,?,?)",         [(r["id"], r["session_id"], serialize_f32(v)) for r, v in zip(rows, vectors)],     )     conn.commit() ``` Note: vec0 has no UPSERT — to re-embed an edited row, `DELETE` then `INSERT` (helper `reembed_one()` does this).  *3) Hybrid BM25 × cosine — Reciprocal Rank Fusion* ```python # fuse by RANK, not raw score (incomparable scales). higher fused = better. for rank, r in enumerate(fts_rows, 1):  scores[r.id] += 1/(60+rank) for rank, r in enumerate(vec_rows, 1):  scores[r.id] += 1/(60+rank) # k_rrf=60 is the standard default; sort desc, take top-N ``` `hybrid_search(conn, query_text, query_vec, limit=10, pool=50)` pulls top-`pool` from each side, fuses, returns merged rows with `rrf_score` + per-side ranks. `hybrid_search_weighted(...)` is the min-max blended alternative with tunable `w_text`/`w_vec`.

*Bootstrap + gotchas* ```python conn = init_all("journal.db", dim=384)   # loads sqlite-vec, creates FTS+vec, rebuilds FTS backfill_embeddings(conn, embed_fn)      # then fill embeddings once ``` Watch-outs from experience: • *Extension loading:* `enable_load_extension(True)` must wrap `load_extension("./sqlite-vec.so")`; I re-disable it afterward. On some distros the stock Python `sqlite3` is built without `enable_load_extension` — if so you need a pysqlite3-binary or a Python linked against a loadable-extension SQLite. • *Don't put the embedding inside the FTS table* — keep them as two virtual tables joined on id; mixing them kills BM25. • *`k = ?` is required* for vec0 KNN; a bare `LIMIT` won't use the ANN path on large tables. • *Dimension lock:* `FLOAT[384]` is fixed at create time — switching models to 768d means a new vec table + re-backfill. • *External-content FTS:* if you ever bulk-edit `messages` outside these triggers, run the `'rebuild'` command to resync.
